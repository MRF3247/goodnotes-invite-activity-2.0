# Goodnotes极速版 · 推荐计划活动系统 — 功能与实现说明

> 面向：交接/评审/日报存档。描述系统「做了什么」「怎么实现」「复杂在哪」。

---

## 0. 一句话概述

一个「老带新」邀请拉新活动的完整 Web 系统：用户报名领邀请码、邀请好友兑换会员、实时排行榜冲榜、按名次/达标发奖；配套后台做账号与邀请码管理、激励号（马甲）氛围、风控、发奖导出。数据由「飞书文档 + Claude Code」每小时清洗后回推，前台展示。

**技术栈**：单进程 Node.js 22 + 内置 SQLite（`node:sqlite`），**零第三方依赖**；纯服务端渲染 + 原生 JS 前端；Docker + K8s 部署，密钥经 k8s Secret 注环境变量。

## 1. 整体架构与数据流

```
                    ┌─────────────── 前台单页 (index.html) ───────────────┐
用户 ── App Banner ─▶│ 报名 / 登录 / 排行榜 / 我的战绩 / 领奖              │
                    └───────┬───────────────────────────────▲────────────┘
                            │ HTTP JSON                       │
                    ┌───────▼───────────────────────────────┴────────────┐
                    │  Node 服务 (server.js 路由分发)                       │
                    │  routes/public.js  routes/ingest.js  routes/admin.js │
                    │  lib: phase / rewards / snapshot / risk / guard ...  │
                    └───────┬──────────────────────────┬──────────────────┘
                            │                           │
                    ┌───────▼────────┐          ┌───────▼────────┐
                    │ SQLite data.db │          │ 定时调度器      │
                    │ (9 张表)        │          │ (每分钟 tick)   │
                    └────────────────┘          └────────────────┘
                            ▲
                            │ X-Ingest-Key（3 个接口）
              ┌─────────────┴──────────────┐
              │ Claude Code 例程（每小时）   │  读飞书表B → 刷码簇风控清洗
              │ 飞书文档(lark-cli)          │  → 按邀请码推「净邀请人数」
              └────────────────────────────┘
```

**核心数据流（每小时）**：后端把兑换明细维护到飞书表 → CC 拉取+风控清洗 → 通过 `/api/ingest/scores` 回推「码→净人数」→ 写入待发布(pending) → 错峰约 30 分钟后自动发布(activations) → 重算榜单快照 → 前台展示。运营也可在后台「立即刷新榜单」手动发布。

## 2. 数据模型（SQLite，9 张表）

| 表 | 作用 | 关键字段 |
|---|---|---|
| `participants` | 报名账号 | uid(唯一)、xhs_id/收件信息(中奖后填)、code(发放的邀请码)、status、**deleted_at(软删除)** |
| `codes` | 邀请码池 | code(主键)、batch_id、activations(净人数)、manual_delta(人工修正)、pending(待发布)、**assigned_to(归属账号)**、**shadow_id(激励号占用)**、max_redemptions、有效期 |
| `shadows` | 激励号(马甲) | nickname、score、auto(自动配分)、code(占用的真实邀请码) |
| `snapshots` | 榜单快照 | snap_date、entry_type(p真实/s激励)、ref_id、rank、score、code(冻结展示) |
| `deletions` | 软删除回收站 | kind(account/code)、participant_id、payload(恢复用JSON)、deleted_at |
| `redemptions` | 兑换明细(风控) | code、user_id、ip、device、redeemed_at |
| `excluded_users` | 风控剔除名单 | user_id、reason |
| `settings` | 运行时配置(KV) | 奖励档位、活动时间、标题标语、阈值等，后台可改即时生效 |
| `logs` | 操作审计 | action、detail、时间 |

启动时自动建表 + **轻量迁移**（`ensureColumn` 补列、必要时重建 participants 表放开旧约束），兼容老库平滑升级。

## 3. 核心业务逻辑（复杂度重点）

### 3.1 活动阶段状态机（`lib/phase.js`）
- 三态：`未开始 / 进行中 / 已结束`。
- 默认 `auto`：按后台设定的**北京时间**开始/结束点自动判定（用 `Intl` 锁 Asia/Shanghai 时区，规避容器 UTC 差 8 小时的坑）；也可手动强制某一态兜底。
- 前台/后台/调度器共用同一判定，避免多处不一致。

### 3.2 邀请码生命周期（较复杂 ★★★★）
一个码可能处于：`未分配(池中)` →（真人报名，进行中按序发放）`归属账号` /（新建激励号）`被激励号占用(shadow_id)` /（删码）`暂扣(可恢复)` /（回收站到期）`释放回池`。
- **报名与发码解耦**：报名只按顺序登记（code 为空），导入的码先单独放着；**活动开始后**（调度器 tick / 用户登录 / 后台切进行中）才**按报名顺序**从池里依次发码。好处：上期用户本期不登录就不占码、不进名单。
- **激励号占真实码**：新建激励号时从池中占一个真码，可被真实兑换（乱兑视为增长），但其邀请人数由算法/人工决定、不走 CC 同步。

### 3.3 奖励计算（`lib/rewards.js`，规则复杂 ★★★★）
- **冲榜奖**（按名次，可配区间，如 4–10 名同奖）+ **达标奖**（拉 N 人得 N 月，只发最高完成一档）。
- **不可叠加**：进入名次奖区间只发冲榜奖，否则发达标奖 —— 前台战绩、后台导出、发奖名单统一按此口径。
- 奖励档位全部存 `settings`(JSON)，**后台可视化增删改、即时上架**，前台三处（战绩/规则/榜单奖品列）同步。

### 3.4 激励号软锚定（`lib/snapshot.js`，算法复杂 ★★★★）
- 自动配分锚定真人榜前列分数（目标名次区间可配），**只涨不跌、整数**，营造竞争氛围又不穿帮；也可切手动钉分。生成快照时统一参与排序。

### 3.5 榜单快照（`runSnapshot`）
- 合并真实账号(激活+人工修正) 与 激励号 → 统一排序、冻结每行展示的邀请码 → 落 `snapshots`。公开榜只取前 10；后台可分页预览全量并标注激励号。

### 3.6 数据同步管线（错峰发布）
- CC 推入 → `codes.pending` → 排期 `publish_delay_min` 分钟 → 调度器到点 `publishPending`（pending 落 activations，保留人工扣分）→ 重算榜单。给运营留反应窗口，也可手动立即发布。

### 3.7 软删除 + 5 天回收站（`deletions` + `purgeDeletions`）
- 删账号 / 删邀请码都是**软删除**：记录恢复所需数据，隐藏效果立即生效，**5 天内可一键恢复**，超期由调度器彻底清除（账号删行+释放码，暂扣码放回池）。

### 3.8 风控（`lib/risk.js`）
- 从兑换明细识别「同 IP / 同设备跨多账号」刷码簇，阈值可配；生成预警报告、支持人工剔除并持久扣分（不被下轮拉取覆盖）。

## 4. 接口清单（分三层）

**A. 用户端（`routes/public.js`，无需登录，部分需图形验证码）**
`GET /api/meta`（阶段+奖励+标题） · `GET /api/captcha` · `POST /api/register`+`/api/claim`（两步报名） · `POST /api/join`（报名/登录二合一） · `POST /api/recover`（登录） · `POST /api/myrank`（查战绩） · `GET /api/leaderboard`（前10） · `POST /api/winner-info`（领奖信息）

**B. 数据对接（`routes/ingest.js`，`X-Ingest-Key` 鉴权，给 CC/后端用）**
`POST /api/ingest/codes`（灌码） · `GET /api/ingest/participants`（报名↔码对应） · `POST /api/ingest/scores`（推净人数）

**C. 后台（`routes/admin.js`，会话 Cookie 鉴权）**
登录 / 总览 / 榜单快照预览(分页) / 立即刷新榜单 / 账号管理(分页) / 邀请码管理(分页) / 删除·恢复·回收站 / 邀请码导入 / 激励号增删改 / 奖励设置 / 活动开关与时间 / 风控导入·报告·剔除 / 查询·人工修正·取消资格 / 导出全量数据(【严禁外传】) / 操作日志。

## 5. 前端形态（`public/index.html` 单页 · 阶段驱动）

| 阶段 | 未登录 | 已登录 |
|---|---|---|
| 未开始 | 简洁报名（只填 UID）→「报名成功，等待开始」 | 同上（仅提示） |
| 进行中 | 排行榜 +「我要报名」+「登录」标签 | 标题下常驻邀请码+推广文案；[排行榜 \| 我的战绩] 浏览器式标签；退出登录 |
| 已结束 |「活动已结束」+ 登录框 | 榜单 + 我的战绩 + 领奖（此时才可填） |

前台仅用原生 JS + 一个通用 `common.js`（api/验证码/复制），标题标语、奖励均由后台配置动态渲染。

## 6. 安全与防刷
- **鉴权分层**：后台会话 Cookie（HMAC 签名、12h）/ 用户免验证码令牌 rankToken（HMAC、30天）/ CC 用 `X-Ingest-Key`；密钥生产走环境变量注入，不入库。
- **防刷**：图形算术验证码、按 IP/UID 频率限额、连续失败阶梯锁定；UID 为唯一身份，格式校验。
- **隐私**：UID/IP/设备/收件信息仅后台可见；导出文件名标【严禁外传】；公开榜只露邀请码不露身份。

---

## 7. 部署运维
- 单进程 `node server.js`；Docker 镜像 + K8s；`config.json`/`data/` 不入库，密钥经 Secret 注入（`ADMIN_TOKEN`/`INGEST_KEY`/`ADMIN_READ_KEY`）。
- 调度器每分钟兜底：定时发布、按序发码、每日刷新快照、清理超期删除。
- 多赛季复用：身份保留、邀请人数清零、旧成绩归档（换新码批次）。

## 8. 功能复杂度评估

| 子系统 | 复杂度 | 复杂点 |
|---|---|---|
| 邀请码生命周期 + 报名发码解耦 | ★★★★ | 码在「池/归属/激励占用/暂扣/释放」多态流转，且发放要按报名顺序、受阶段与库存约束 |
| 奖励体系（不可叠加、可配置） | ★★★★ | 名次奖×达标奖两套规则的优先级与一致性（前台/导出/发奖三处口径统一），且档位后台可配 |
| 激励号软锚定算法 | ★★★★ | 跟随真人分数、只涨不跌、区间控制、占真实码不穿帮 |
| 软删除 + 5 天回收站 | ★★★☆ | 账号/码两类删除各自的隐藏、恢复、到期彻底清除，且不破坏码池与排序 |
| 数据同步错峰管线 | ★★★☆ | pending→延迟发布→重算，人工/无人值守双模式，人工扣分持久 |
| 阶段状态机 + 时区 | ★★★☆ | 北京时区自动判定 + 手动兜底，多处共用 |
| 风控刷码簇识别 | ★★★☆ | 同 IP/设备跨账号聚类、阈值、持久扣分 |
| 前台阶段化单页 | ★★★☆ | 3 阶段 × 登录态的多形态渲染与状态流转 |
| 后台管理面板 | ★★★☆ | 十余个功能标签、分页/跳页、导出、审计 |
| 基础设施（零依赖/迁移/部署） | ★★★☆ | 内置 SQLite、启动自动迁移、Docker/K8s、密钥注入 |

**整体规模**：后端约 10 个模块 + 9 张表 + 三层共约 30 个接口；前台单页阶段机 + 后台十余标签。属**中等偏上复杂度**的完整业务系统（含数据管线、算法氛围、风控、软删除、多赛季复用等非平凡机制）。
