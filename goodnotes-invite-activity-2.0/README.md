# Goodnotes极速版 邀请活动系统

登记页 / 排行榜页 / 后台管理，单进程 Node.js 服务，零外部依赖（无需 npm install）。

## 运行要求

- Node.js **22.5 或更高版本**（使用了内置的 node:sqlite，无需安装任何 npm 包）

## 启动

```bash
node server.js
```

启动后：

| 页面 | 地址 |
|---|---|
| 登记页 | http://localhost:3210/ |
| 排行榜 | http://localhost:3210/leaderboard |
| 后台 | http://localhost:3210/admin |

生产环境建议用 pm2 或 systemd 守护进程运行，并用 Nginx 反向代理到 3210 端口（配 HTTPS）。
如经过反向代理，请让 Nginx 传递 `X-Forwarded-For` 头，否则频率限制会把所有人当成同一个IP。

## 目录说明

```
config.json        端口、后台口令、签名密钥、数据目录 —— 部署前请检查
data/data.db       SQLite 数据库（所有报名数据、邀请码、日志都在这一个文件里）
banned-words.txt   昵称违禁词，每行一个，可随时增删（改后需重启）
public/            前端页面与静态资源
```

**升级版本时：覆盖代码文件即可，切勿删除 `data/` 目录**（那是全部活动数据）。
迁移服务器时：整个文件夹打包带走即可，数据在 `data/data.db` 里。

数据库无需安装和初始化：首次启动服务时自动创建 `data/data.db` 及全部表结构。
**建议活动期间每天备份一次**（cron 定时拷贝该文件即可）：
`0 3 * * * cp /opt/goodnotes-invite-activity/data/data.db /opt/backups/data-$(date +\%F).db`

## config.json 字段

| 字段 | 说明 |
|---|---|
| port | 服务端口，默认 3210 |
| adminToken | 后台登录口令（交付时已随机生成，可自行更换） |
| secret | 签名密钥（验证码/会话用），部署时建议换一个新的随机值 |
| dataDir | 数据目录，默认 ./data |

其余运营配置（名额上限、活动起止日期、榜单更新时间、频率限制、激励号区间等）都在**后台 → 活动设置**里改，改完即时生效，不用重启。

## 上线前检查清单

- [ ] config.json 里换掉 adminToken 和 secret（`node -e "console.log(require('crypto').randomBytes(24).toString('base64url'))"` 可生成）
- [ ] 后台设置好活动开始/结束日期、名额上限
- [ ] 导入正式邀请码（后台 → 邀请码，单列CSV，数量应 ≥ 名额上限；生效期由后端系统管理）
- [ ] 确认 data/data.db 是干净的（测试数据请勿带上生产）
- [ ] Nginx 配好 HTTPS 和 X-Forwarded-For
- [ ] **向App开发确认：App内置浏览器（WebView）是否开启 localStorage/DOM Storage**。
      排行榜"免输入自动识别身份"依赖它（Android WebView 默认可能关闭）。
      不支持时功能自动降级为手动输入UID+小红书号查询，不影响活动进行，但体验有差别，建议开启
- [ ] 修改了 public/ 下的 CSS/JS 后，记得把三个 HTML 里的 `?v=2` 版本号加一，否则用户可能看到缓存的旧版

## 每日运营流程（人工审核制）

排行榜不自动更新，由运营每天审核后手动发布：

1. 从后端拿到两张表：**表1 兑换明细**（code,userId）、**表2 用户设备**（userId,ip,deviceId）
2. 后台 → 数据与风控 → ① 导入两张表（全量覆盖）
3. ② 生成风控报告：检查同设备多UID（强信号，直接违规）、同IP聚集（弱信号，
   注意宿舍/公司WiFi误报）、重复兑换UID；把确认违规的UID填入剔除框
4. ③ 点「计算积分并发布榜单」：按剔除后的净数据（每码=去重UID数）刷新排行榜

预警阈值、更新频率文案在后台「活动设置」中调整。应急时可用「手动覆盖激活次数」
直接导入（邀请码,次数）两列CSV。每日 00:00 有一次兜底榜单刷新（不改变积分）。

## 给后端同事的需求（邀请码规格）

- 邀请码**无兑换总次数上限**（一个码可被无限多人兑换，不能设"仅N次有效"）
- **需要合理的单位时间频率限制**：防止异常刷码（如1分钟上千次）冲击服务，
  具体阈值请后端按服务承载和风控评估——但不能误伤正常用户
  （正常场景：一个码一天被兑换几十次已是头部水平）
- 仅可邀请新用户：若对方账户存在生效中的会期，则无法使用
- 数量 ≥ 报名名额上限（默认9999），建议留余量
- 活动期间需要能按邀请码导出/查询累计激活次数
